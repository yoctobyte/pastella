## 1. The goal, precisely

**Two or three laptops, in different physical places, gossiping sensor readings into
one realm, over the real internet, for a week.**

Done when:

- 3 nodes at ≥2 physical locations converge on the same readings;
- it survives a closed lid, a dropped link, a changed IP, a process restart;
- readings are **signed** and verified; a wrong-realm peer is **rejected at connect**;
- it runs a week unattended.

**Not** a demo, not a product. The first *running* network.

Usage pattern (copy from `src/test_realm.pas:28,57`):

```pascal
if await RealmHandshake(fd, realmKey, {initiator:} True, sess) then
  await SealSend(fd, sess, payload);
```

A wrong key makes `RealmHandshake` return `False`. **That is the whole membership
check for v0.1.**

### `src/pastella_node.pas` (238 lines) — the thing to REPLACE

Today: coroutine reactor + PAL sockets + offer/fetch. But **no realms, no signing**,
localhost ports only, fixed round count, then exits. Keep its reactor shape; replace
its guts.

### `src/gossip_tcp.pas` / `src/gossip_mem.pas` — the anti-entropy core

`HAVE` (offer hashes) → `WANT` (ask for missing) → `DATA` (send objects). Content
addressed: an object's name is `Sha256Hex(Sha256(bytes))`. Keep this exactly.

### `src/test_discovery.pas` — LAN beacons, already working (`RealmBeacon`/`BeaconValid`).

### From frank2's RTL (already shipped, already tested)

```pascal
uses sysutils, sha256, ecdsa_p256, platform, scheduler;

function EcdsaP256GenKey(var priv, pub: AnsiString): Boolean;   { priv 32B, pub 64B }

function EcdsaP256Verify(const qxy, msg, sig: AnsiString): Boolean;
```

Signing costs ~12 ms, verify ~19 ms. Fast enough to sign every reading.

## 3. What to build

### 3.1 The object envelope — `src/object.inc` (NEW)

The store holds **bare blobs** today (`TItem = record hash, data end`). Readings must
be signed or "verified" is a lie. Minimal envelope — **nothing speculative**:

```
object bytes (all fields length-prefixed with a u16, so parsing is trivial):
  ver      : u8         = 1
  otype    : u8         = 1 (READING)     ; room for more later
  realmID  : bytes      ; which realm — reject anything else
  author   : bytes(64)  ; the author's ECDSA-P256 public key
  payload  : bytes      ; type-specific (§3.2)
  sig      : bytes(64)  ; EcdsaP256Sign(authorPriv, everything above)
```

```pascal
function  ObjEncode(otype: Integer; const realmID, authorPub, payload, authorPriv: AnsiString): AnsiString;
function  ObjVerify(const raw, realmID: AnsiString;
                    var otype: Integer; var authorPub, payload: AnsiString): Boolean;
```

`ObjVerify` returns False if: version unknown, realmID mismatch, signature bad, or the
bytes are malformed. **Verify BEFORE storing. Never store or relay an object that
fails.**

**Deliberately NOT included** (do not add them "while you are there"):
`refs`/DAG, epochs, `created` timestamps in the envelope, encryption of the payload
(the *transport* is already sealed by `SealSend`).

### 3.2 The READING payload

```
  sensor : bytes    ; free text, e.g. "office-temp"
  t      : u64      ; unix ms, from the producer (ADVISORY — display only)
  value  : i32      ; milli-degrees, e.g. 21500 = 21.5 C
```

v0.1's sensor is **fake**: a slow sine wave, one reading every 30 s. Real hardware
later — it changes nothing above.

### 3.3 The node — `src/pastella_node.pas` (REWRITE)

```
usage: pastella_node <config-file>
```

**Config file** (plain text, one `key = value` per line — no parser library needed):

```
realm      = my-secret-passphrase      # -> RealmKey(passphrase)
listen     = 0.0.0.0:34711
peer       = home.example.org:34711    # repeatable; may be absent
peer       = 10.0.0.5:34711
lan        = on                        # LAN beacons on/off
sensor     = office-temp               # '' = viewer only, produce nothing
datadir    = ~/.pastella/office
```

**Behaviour:**

- **listen**: accept connections; on each, `RealmHandshake(fd, key, False, sess)`.
  Fail → close. Then serve HAVE/WANT/DATA over `SealSend`/`SealRecv`.
- **dial**: for each configured/cached peer, a coroutine that connects, handshakes
  (`initiator=True`), gossips, and **loops forever with backoff** (see §3.5).
- **produce**: if `sensor` is set, every 30 s build a READING, `ObjEncode` it, store
  it. It propagates by ordinary gossip — no special path.
- **view**: print every new reading as it arrives:
  `[2026-07-13 09:31:02] office-temp = 21.5 C  (from 3a9f…)`

### 3.4 Persistence — `datadir/`

```
datadir/
  key.priv          # 32B ECDSA private key — generated on first run, chmod 600
  key.pub           # 64B
  peers.txt         # host:port, one per line. Learned + configured. Append on success
  objects/<hash>    # one file per object, raw bytes. The store IS the directory
```

Store = a directory. Startup = read it. **No database. No index.** At v0.1 scale
(a few thousand tiny objects) a linear scan is correct and fast, and it cannot corrupt.

### 3.5 Robustness — the part that actually matters

This is what makes it a network rather than a demo:

- **Never exit.** No round count. The node runs until killed.
- **Dial loop, per peer**: connect → handshake → gossip until the socket dies → sleep
  → retry. Backoff **1s, 2s, 4s … capped at 60s**; reset on success.
- **A dead peer is normal**, not an error. Log once, keep retrying, never crash.
- **Re-resolve DNS on every dial attempt** — this is what survives a changed IP.
- **Idle keepalive**: gossip round every 30 s even when nothing is new (cheap: an empty
  HAVE), so a dead link is *noticed*.
- **Duplicate objects are free** (content-addressed) — dedupe by hash, never by
  sequence.

## 6. Non-goals — say no

Not in v0.1. If one seems needed, **write it down and carry on without it**:

DHT · NAT traversal / hole-punching · relays · mailboxes / offline delivery · trust
graph · attestations · admission tiers (invite/approve/open) · key epochs · membership
log · 1:1 · file sharing · chunking / GC · BitTorrent piggyback · encrypted rendezvous
blobs · anchors · profiles · the object DAG (`refs`) · ESP32.

**Bootstrap for v0.1 = manual `peer =` lines + LAN beacons. That is all.**

**Why no NAT work:** outbound connections traverse NAT for free. One reachable laptop
(home, fixed IP or dynamic DNS); the others dial *out* to it. Star topology is fine —
the reachable node is a **member**, not a relay.

## 7. Known traps

- **pxx dialect:** a managed-record-returning call must not be passed directly as an
  argument to another call — bind it to a temp first. (Bites `AnsiString`-heavy code;
  see the note at the top of frank2's `ecdsa_p256.pas`.)
- **Rebuild before believing a test.** A stale binary silently passes. `rm` the output
  or grep it for a new string.
- **`build.sh` uses frank2's PINNED compiler** (`stable_linux_amd64/default/pinned`),
  not a local build. If a pxx feature seems missing, the pin may predate it.
- **riscv32/ESP32 is broken** at the compiler level right now (writes through a `var`
  array parameter are lost or crash — frank2 `bug-riscv32-p256field-coredump`).
  Irrelevant to v0.1 (laptops only), and it is *not our job to fix*.
