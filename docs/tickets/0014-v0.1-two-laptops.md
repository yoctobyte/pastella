# 0014 — v0.1: the smallest real network

- **Status:** open — **this is the only thing that matters right now.**
- **Depends:** nothing. Everything it needs already exists in `src/`.

---

## The MVP, in one line

**An ESP32-capable, shared-realm-key sensor network.**

- **shared realm key (PSK)** — objects are authenticated with an **HMAC-SHA256 under
  the realm key**, *not* a per-author ECDSA signature. Fast on a chip (which has a SHA
  accelerator), and it deletes the 1–2 s/signature problem entirely.
  **Honest cost:** with a shared key, *any member can forge another member's readings*.
  For a mesh of your own devices that is fine. Per-author attribution needs asymmetric
  identity (the CA-realm path) and is a later upgrade, not an MVP requirement.
- **ESP32-capable** — the design must not preclude the chip: bounded RAM, no full DAG,
  small objects, no ECDSA in the hot path, UDP-ready
  ([0015](0015-udp-datagram-transport.md)).
- **but built on laptops first** — see below.

## ⚠️ The ESP32 leg is BLOCKED, and not by us

Measured 2026-07-13 with frank2's existing `lib_*` suites:

| | riscv32 (ESP32-C3) | arm32 | x86-64 / aarch64 |
|---|---|---|---|
| **SHA-256** | **core-dump** | ok | ok |
| **ChaCha20-Poly1305** | **core-dump** | **core-dump** | ok |

**An ESP32 cannot currently compute a SHA-256.** Not a signature — a *hash*. So HMAC,
HKDF and the sealed transport are all unavailable on the chip, **independently of
ECDSA**. (frank2 `bug-riscv32-p256field-coredump`, whose title badly understates it:
the suspected root cause is that *writes through a `var` array parameter* are broken on
the 32-bit backends, and SHA/ChaCha are exactly that shape.)

`examples/esp32/net-c3` proves **networking** works on device. It does not extend to
anything **sealed or authenticated**.

**This is a frank2 problem on another track, and it is not ours to fix.** It will be
fixed sooner or later.

**So: laptops first, and the plan below is unchanged.** The MVP definition stands; its
ESP32 leg simply waits. A laptop is a leaf node with a bigger battery, and it debugs
better anyway.

## The goal, deliberately small

**Two or three of my own laptops, in different physical places, gossiping readings
into one realm across the internet. Running for days. That is all.**

Not a demo. Not a product. A **running network** — the first one — small enough that
every failure is visible and diagnosable.

## Why this and nothing else

Twelve tickets and six design documents were written before a single new line of
protocol code. **Designs that never meet an implementation are always right, and
always useless.** This ticket is the corrective.

Everything else in `docs/` is **thinking, not commitment**. It stays on the shelf
until this runs.

## Honest status: the pieces are proven; the NODE is not built

Every capability is proven **in isolation, by its own test** — realms here, gossip
there, discovery elsewhere. **Nothing is proven together.** `pastella_node.pas` (238
lines) is a loopback demo, not a node:

- it does **not use realms** (no handshake, no sealing — `realm.inc` is only exercised
  by `test_realm`);
- it does **not sign anything** (the store is still untyped blobs: `hash, data`);
- it takes **localhost ports**, not real hosts, and runs a **fixed number of rounds**
  then exits.

The whole codebase is **~1,900 lines**. That is the good news: **v0.1 is glue, not a
build.**

## The work list

1. **Real peers.** `host:port` from config/CLI, not localhost ports. Plus a persistent
   peer cache so a restart is not a re-setup.
2. **Realm handshake on the connection path.** `realm.inc` exists; wire it into the
   node so a wrong-realm peer is rejected at connect.
3. **Authenticated object envelope.** The store holds bare blobs today. Readings must
   be authenticated and realm-scoped, or "verified" means nothing.
   **Use an HMAC-SHA256 under the realm key — NOT ECDSA** (see the MVP definition
   above: this is what makes it ESP32-viable). Envelope = `ver | otype | realmID |
   payload | mac`. No `author`, no keypair, no signature. Skip `refs`/epochs until
   something needs them.
4. **A `READING` type**, a fake sensor producing them, and a viewer. `WriteLn` is a
   viewer.
5. **Persistence.** Store + peer cache survive a restart.
6. **Robustness.** Reconnect loops instead of a fixed round count; survive a dropped
   link, a closed lid, a changed IP.

**Bootstrap, in the dumbest form that works:** manual `add-peer <host:port>`, plus the
LAN beacon that already exists. That is *it*. No DHT, no NAT traversal, no relays, no
anchors, no encrypted rendezvous blob — those are
[0013](0013-bootstrap-profiles-and-rendezvous.md)'s menu, and **the menu is not the
meal.** Which channel an application uses is an application choice; v0.1 chooses the
two cheapest.

## What makes it work with no NAT code at all

**Outbound connections traverse NAT for free.** One laptop that stays put (home, a
fixed IP or dynamic DNS) is enough: the others connect *outward* to it and gossip.
Star topology, and that is fine — the reachable node is a **member**, holding its own
realm's content.

So: **no hole-punching, no relay, no DHT.** One reachable machine.

## Explicitly NOT in v0.1

Everything else. Named, so it stays out:

DHT · NAT traversal · relays · mailboxes · trust graph · attestations · admission
tiers · key epochs · membership log · 1:1 · file sharing · the `open` tier · BitTorrent
piggyback · anchors · profiles.

**Each of those is a hypothesis. v0.1 is the experiment that tells us which ones we
actually need** — and it will be fewer than we think.

## Acceptance

- Three nodes, at **two or more physical locations**, converge on the same readings.
- It survives: a laptop closing its lid, a network dropping, an IP changing, a
  process restart.
- It runs for **a week** without intervention.
- Readings are signed and verified; a wrong-realm peer is rejected.
- **We learn something that contradicts the design docs.** *(If nothing in `docs/`
  turns out to be wrong, we were not paying attention.)*

## ESP32

**Later.** The net stack works on device; the crypto does not — and the riscv32
backend has a bug far wider than crypto (writes through a `var` array param are
broken for any element type; frank2 `bug-riscv32-p256field-coredump`). It will be
fixed sooner or later, and that is not our problem right now.

Laptops first. A laptop *is* a leaf node with a bigger battery.
